    # Feature Specification: 수강신청 시스템 (Course Registration System)

**Feature Branch**: `001-course-registration`
**Created**: 2026-02-08
**Status**: Draft
**Input**: 고성능 수강신청 시스템 - 동시성 제어를 통한 정원 초과 방지

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 수강신청 (Priority: P1)

학생이 원하는 강좌를 신청하여 해당 학기 시간표에 추가한다. 수강신청 기간 동안 많은 학생이 동시에 접속하더라도 시스템은 정확하게 정원을 관리하고, 선착순으로 신청을 처리한다.

**Why this priority**: 수강신청 시스템의 핵심 기능이며, 동시성 제어가 가장 중요한 영역이다. 정원 초과(Overbooking)가 발생하면 시스템의 신뢰성이 무너진다.

**Independent Test**: 학생 계정으로 로그인 후 강좌 목록에서 원하는 강좌를 선택하여 신청하고, 시간표에 반영되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 정원이 남아있는 강좌가 있고 학생이 해당 강좌를 신청하지 않은 상태, **When** 학생이 해당 강좌를 신청하면, **Then** 신청이 성공하고 강좌가 학생의 시간표에 추가되며 현재 신청 인원이 1 증가한다
2. **Given** 정원이 1명 남은 강좌가 있고, **When** 100명의 학생이 동시에 신청하면, **Then** 정확히 1명만 성공하고 나머지 99명은 정원 초과 메시지를 받는다
3. **Given** 학생이 이미 17학점을 신청한 상태, **When** 3학점 강좌를 신청하면, **Then** 최대 학점(18) 초과로 신청이 거부된다
4. **Given** 학생이 월요일 10시-12시 강좌를 이미 신청한 상태, **When** 월요일 11시-13시 강좌를 신청하면, **Then** 시간 중복으로 신청이 거부된다
5. **Given** 학생이 이미 특정 강좌를 신청한 상태, **When** 같은 강좌를 다시 신청하면, **Then** 중복 신청 불가 메시지를 받는다

---

### User Story 2 - 수강취소 (Priority: P2)

학생이 이미 신청한 강좌를 취소하여 시간표에서 제거하고, 해당 강좌의 정원을 다시 확보할 수 있다.

**Why this priority**: 수강신청 다음으로 중요한 기능으로, 학생이 잘못 신청하거나 마음이 바뀌었을 때 필수적이다. 취소된 정원은 다른 학생에게 기회가 된다.

**Independent Test**: 시간표에서 기존 신청 강좌를 선택하여 취소하고, 시간표에서 제거되며 강좌 정원이 회복되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 학생이 특정 강좌를 신청한 상태, **When** 해당 강좌를 취소하면, **Then** 취소가 성공하고 강좌가 시간표에서 제거되며 현재 신청 인원이 1 감소한다
2. **Given** 학생이 신청하지 않은 강좌, **When** 해당 강좌를 취소하려고 하면, **Then** 신청 내역이 없다는 오류 메시지를 받는다

---

### User Story 3 - 강좌 목록 조회 (Priority: P3)

학생이 이번 학기에 개설된 강좌 목록을 조회하여 수강신청할 강좌를 탐색한다.

**Why this priority**: 수강신청 전 필수적인 탐색 기능이며, 학생이 올바른 선택을 할 수 있도록 충분한 정보를 제공해야 한다.

**Independent Test**: 강좌 목록 페이지에서 전체 강좌 조회가 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 시스템에 강좌가 등록되어 있고, **When** 전체 강좌 목록을 조회하면, **Then** 모든 강좌가 강좌명, 교수명, 학점, 강의 시간, 전체 정원, 현재 신청 인원 정보와 함께 표시된다

*Note: 학과별 필터링 기능은 추후 비즈니스 규칙 정의 후 추가 예정*

---

### User Story 4 - 내 시간표 조회 (Priority: P4)

학생이 자신이 신청한 강좌 목록을 시간표 형태로 조회하여 이번 학기 수업 일정을 확인한다.

**Why this priority**: 학생이 자신의 수강 현황을 파악하는 데 필수적이며, 추가 신청이나 취소 결정에 도움을 준다.

**Independent Test**: 로그인한 학생의 시간표 페이지에서 신청된 강좌들이 올바르게 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 학생이 여러 강좌를 신청한 상태, **When** 시간표를 조회하면, **Then** 신청한 모든 강좌가 요일/시간별로 정리되어 표시된다
2. **Given** 학생이 아무 강좌도 신청하지 않은 상태, **When** 시간표를 조회하면, **Then** 빈 시간표와 함께 안내 메시지가 표시된다

---

### User Story 5 - 학생/교수 목록 조회 (Priority: P5)

관리자 또는 시스템 사용자가 등록된 학생 및 교수 정보를 조회할 수 있다.

**Why this priority**: 시스템 관리 및 데이터 확인을 위한 보조 기능으로, 핵심 수강신청 기능보다 우선순위가 낮다.

**Independent Test**: 학생 목록과 교수 목록 조회 기능이 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 시스템에 학생이 등록되어 있고, **When** 학생 목록을 조회하면, **Then** 모든 학생의 기본 정보가 표시된다
2. **Given** 시스템에 교수가 등록되어 있고, **When** 교수 목록을 조회하면, **Then** 모든 교수의 기본 정보가 표시된다

---

### Edge Cases

- 정원이 0인 강좌(개설 취소된 강좌)에 대한 신청 시도 시 적절한 오류 메시지 반환
- 존재하지 않는 학생 ID나 강좌 ID로 신청 시도 시 적절한 오류 처리
- 수강신청 기간 외에 신청 시도 시 기간 외 메시지 반환 (기간 관리가 구현된 경우)
- 동일 강좌에 대한 동시 신청과 취소가 발생할 때 데이터 정합성 유지
- 네트워크 장애나 시스템 오류 발생 시 부분 처리된 트랜잭션 롤백
- 정원 초과로 신청 실패 시: 실패 메시지만 반환하며 대기열 기능 없음. 사용자가 직접 재시도해야 함

## Requirements *(mandatory)*

### Functional Requirements

#### 학생 관련
- **FR-001**: 시스템은 등록된 학생 목록을 조회할 수 있어야 한다
- **FR-002**: 학생은 자신의 이번 학기 시간표(신청한 강좌 목록)를 조회할 수 있어야 한다

#### 강좌 관련
- **FR-003**: 시스템은 전체 강좌 목록을 조회할 수 있어야 한다
- **FR-004**: 시스템은 학과별로 강좌를 필터링하여 조회할 수 있어야 한다(정원/현재 인원/시간 정보 포함)
- **FR-005**: 각 강좌 정보에는 강좌명, 교수명, 학점, 강의 시간, 전체 정원, 현재 신청 인원이 포함되어야 한다

#### 교수 관련
- **FR-006**: 시스템은 등록된 교수 목록을 조회할 수 있어야 한다

#### 수강신청 관련
- **FR-007**: 학생은 원하는 강좌를 수강신청할 수 있어야 한다
- **FR-008**: 학생은 신청한 강좌를 수강취소할 수 있어야 한다
- **FR-009**: 시스템은 동시에 여러 학생이 같은 강좌를 신청해도 정원을 초과하지 않도록 보장해야 한다
- **FR-010**: 정원이 1명 남은 상태에서 다수가 동시에 신청해도 정확히 1명만 성공해야 한다

#### 비즈니스 규칙
- **FR-011**: 학생당 한 학기 최대 18학점까지만 신청 가능해야 한다
- **FR-012**: 이미 신청한 강좌와 강의 시간이 1분이라도 겹치는 강좌는 신청이 거부되어야 한다. 단, 경계 시간(예: 11:00 종료 ↔ 11:00 시작)은 중복으로 간주하지 않는다
- **FR-013**: 현재 수강 인원이 정원에 도달한 강좌는 신청이 거부되어야 한다
- **FR-014**: 이미 신청한 강좌를 다시 신청하면 거부되어야 한다

### Key Entities

- **Student (학생)**: 수강신청의 주체. 학번, 이름, 소속 학과, 현재 신청 학점 정보를 가진다
- **Professor (교수)**: 강좌를 담당하는 교수진. 이름, 소속 학과 정보를 가진다
- **Course (강좌)**: 수강신청의 대상. 강좌명, 담당 교수, 학점, 정원, 현재 신청 인원, 강의 시간 정보를 가진다. 강의 시간은 다중 시간대를 지원하며 (예: 월 10:00-11:30, 수 14:00-15:30), 요일별로 상이한 시간대가 허용된다
- **Enrollment (수강신청)**: 학생과 강좌 간의 신청 관계. 신청 일시 정보를 포함한다

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 정원이 1명 남은 상황에서 100명이 동시에 신청해도 정확히 1명만 성공하고 정원 초과가 발생하지 않는다
- **SC-002**: 수강신청 요청 시 사용자는 3초 이내에 성공/실패 응답을 받는다
- **SC-003**: 시스템은 1,000명 이상의 동시 사용자 접속을 처리할 수 있다
- **SC-004**: 학점 제한(18학점) 및 시간 중복 검사가 100% 정확하게 동작한다
- **SC-005**: 신청 성공 시 시간표와 강좌 현황에 즉시 반영된다
- **SC-006**: 시스템 장애 시에도 신청 데이터가 유실되거나 부정확해지지 않는다

## Clarifications

### Session 2026-02-08

- Q: 하나의 강좌가 여러 시간대에 수업을 가질 수 있는가? → A: 다중 시간대 지원 (예: 월 10:00-11:30, 수 14:00-15:30). 요일별로 상이한 시간대 허용
- Q: 정원 초과로 신청 실패 시 사용자에게 제공할 옵션은? → A: 단순 실패 메시지 반환 (대기열 없음, 사용자가 직접 재시도)
- Q: 시간 중복 판정 기준은? → A: 1분이라도 겹치면 중복. 경계 시간은 허용 (10:00-11:00 ↔ 11:00-12:00는 허용)
- Q: 강좌의 학과 필터링 기준은? → A: 현재 범위에서 학과/교수 필터링 제외. 추후 비즈니스 규칙 정의 예정

## Assumptions

- 학생, 교수, 강좌 데이터는 사전에 등록되어 있다고 가정한다 (본 기능 범위에서 등록/수정/삭제는 제외)
- 학생 인증은 별도 인증 시스템을 통해 처리된다고 가정한다
- 수강신청 기간 관리는 본 기능 범위에 포함되지 않는다 (항상 신청 가능 상태)
- 강의 시간은 요일과 시작/종료 시간으로 표현되며, 주간 반복 스케줄로 가정한다
